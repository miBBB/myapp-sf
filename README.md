Документация для разработчиков

1) Введение

Этот документ описывает правила работы с кодовой базой нашего приложения: где находится репозиторий, как вносить изменения, как именовать ветки и как настроить рабочее окружение. Документ обязателен к прочтению для всех новых сотрудников.



2) Где располагается репозиторий

Код приложения хранится в приватном репозитории на GitHub (план Team), доступ к которому ограничен и предоставляется только участникам команды.

    • URL репозитория: https://github.com/miBBB/myapp-sf (заменить на реальный)
    • Доступ выдаёт тимлид/администратор организации по запросу.
    • Без приглашения в организацию репозиторий не виден.

Как посмотреть существующий код:

    1. Получите приглашение в организацию на GitHub и примите его.
    2. Установите Git
    3. Настройте SSH-ключ и добавьте его в свой профиль GitHub.
    4. Склонируйте репозиторий:

```bash
git clone git@github.com:miBBB/myapp-sf.git
cd myapp-sf
```

    5. Просмотреть код можно как локально (в IDE), так и прямо в веб-интерфейсе GitHub.



3) Структура веток

Ветка           Назначение
main            Стабильная продакшн-версия. Прямой push запрещён, изменения только через Pull Request.
develop         Интеграционная ветка, куда вливаются готовые задачи перед релизом.
рабочие ветки   Создаются от develop под конкретную задачу.



4) Процесс внесения изменений (Flow)

Мы используем модель на основе feature-веток с обязательным code review.

    6. Обновите локальную ветку develop.
    7. Создайте свою рабочую ветку от develop.
    8. Внесите изменения, делайте небольшие осмысленные коммиты.
    9. Отправьте ветку на сервер (push).
    10. Откройте Pull Request (PR) в ветку develop.
    11. Дождитесь прохождения CI и ревью минимум одного коллеги.
    12. После одобрения ветка сливается в develop (Squash and merge).
    13. Релиз в продакшн: develop -> main через отдельный PR.

Прямой push в main и develop запрещён настройками branch protection.



5) Правила именования веток

Формат: <тип>/<краткое-описание-через-дефис>

Тип         Назначение                  Пример
feature/    Новая функциональность      feature/user-login
fix/        Исправление бага            fix/header-overflow
hotfix/     Срочная правка в продакшн   hotfix/payment-crash
refactor/   Рефакторинг кода            refactor/api-client
docs/       Документация                docs/readme-update


Рекомендуется добавлять номер задачи из трекера: feature/TASK-123-user-login.

Правила:
    • только латиница, нижний регистр, слова через дефис;
    • без пробелов и кириллицы;
    • название должно отражать суть задачи.



6) Настройка среды разработки

    14. Установите необходимые инструменты (Node.js / Python / и т.д. — согласно стеку проекта).
    15. Установите зависимости проекта.
    16. Скопируйте файл окружения и заполните переменные:

```bash
cp .env.example .env
```

    17. Используйте единый конфиг проекта (линтер, форматтер, настройки редактора). Конфигурация лежит в корне репозитория:

    • Ссылка на конфиг: https://github.com/our-company/our-app/blob/develop/.editorconfig
    • Также см. .eslintrc, .prettierrc и .vscode/settings.json в корне проекта.

Перед началом работы убедитесь, что ваш редактор подхватил .editorconfig — это гарантирует единый стиль кода в команде.



7) Инструкция для младших сотрудников: слияние веток

Шаг 1. Создать свою ветку

```bash
Перейти на develop и обновить его
git checkout develop
git pull origin develop

Создать новую ветку и переключиться на неё
git checkout -b feature/my-new-feature
```

Шаг 2. Внести изменения в ветку

```bash
После редактирования файлов посмотреть статус
git status

Добавить изменения в индекс
git add .

Зафиксировать изменения с понятным сообщением
git commit -m "feat: добавил форму авторизации"

Отправить ветку на сервер
git push -u origin feature/my-new-feature
```

Шаг 3. Слить изменения с основной кодовой базой

    18. Откройте репозиторий на GitHub — появится предложение создать Pull Request.
    19. Создайте PR из feature/my-new-feature в develop.
    20. Назначьте ревьюера и дождитесь одобрения и зелёного CI.
    21. После аппрува нажмите Squash and merge.

Локальная синхронизация после слияния:

```bash
git checkout develop
git pull origin develop

Удалить локальную ветку, которая больше не нужна
git branch -d feature/my-new-feature
```

⚠️ Никогда не сливайте код в main/develop напрямую командой git merge локально — все изменения проходят только через Pull Request и review.



8) Полезные команды

```bash
git log --oneline --graph   # история коммитов
git diff                    # посмотреть несохранённые изменения
git stash                   # временно спрятать изменения
git fetch --prune           # обновить список веток
```
